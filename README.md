# HelpKart — Low-Latency AI Voice Support Agent with RAG
Live Demo:https://helpkart-agent.onrender.com/

A real-time AI customer support agent for HelpKart, built with Node.js, TypeScript, WebSockets, Groq (LLaMA 3.3), Supabase, and a RAG pipeline.

---

## Architecture Overview
```
User (Browser)
    ↕ WebSocket (real-time, bidirectional streaming)
Backend (Node.js / TypeScript / Express)
    ├── Session Manager     → tracks conversation state in memory + Supabase
    ├── RAG Pipeline        → keyword-based retrieval from Supabase knowledge base
    ├── Groq API            → streams responses token by token (LLaMA 3.3 70B)
    └── Supabase (PostgreSQL) → customers, orders, knowledge base, sessions, messages
```

- `orders(customer_id)` — fast order lookups per customer
- `messages(session_id)` — fast message history retrieval per session

## Low-Latency Design Decisions

### 1. WebSockets over HTTP polling
WebSockets provide a persistent, bidirectional connection between client and server. This eliminates the overhead of repeated HTTP handshakes and allows the server to push tokens to the client the moment they are generated. Polling-based approaches would add 200–500ms of unnecessary latency per turn.

### 2. Token-by-token streaming
Instead of waiting for the full AI response before sending it to the client, we stream each token as soon as it is generated by Groq. This means the user sees the first word of the response in under 300ms, making the conversation feel alive and natural.

### 3. In-memory session store
Active sessions are stored in a `Map` in server memory rather than being fetched from the database on every message. This eliminates a DB round-trip on every turn, reducing per-message latency significantly.

### 4. Parallel execution with Promise.all
RAG retrieval and session data preparation are executed in parallel using `Promise.all`, so retrieval does not block or delay the start of the AI response stream.

### 5. Minimal context injection
Only the top 3 most relevant knowledge base chunks are injected into the system prompt. This keeps the prompt short, reduces token count, and speeds up model response time.

## RAG Implementation

### How it works
1. User sends a message
2. `rag.service.ts` extracts keywords from the message
3. Supabase is queried using `ilike` for matching knowledge base entries
4. Top 3 matching chunks are retrieved
5. Chunks are formatted and injected into the system prompt
6. Groq generates a response grounded in the retrieved knowledge

### Why keyword-based retrieval
For a customer support use case with a small, well-defined knowledge base (return policy, shipping, cancellation, etc.), keyword-based retrieval is fast, deterministic, and requires no embedding model or vector similarity computation. This keeps retrieval latency under 50ms.

### Hallucination prevention
The system prompt explicitly instructs the agent: "If you don't know something, say so honestly — never make up information." Retrieved knowledge is clearly labelled so the model knows what it can and cannot reference.

## AI Context & State Management

### Rolling context window
Each session stores up to 10 messages in memory. When the conversation exceeds 10 messages, older messages are trimmed and replaced with a summary, keeping the context window short and response latency stable over long conversations.

### Summarization strategy
When context is compressed, the oldest messages are summarized into 2-3 sentences and stored both in memory and in Supabase. The summary is prepended to the context so the agent retains awareness of earlier conversation topics.

### What happens when conversation gets long
Without compression, a long conversation would pass thousands of tokens to the model on every turn, increasing latency linearly. With rolling compression, the context stays bounded at ~10 messages regardless of conversation length, keeping latency flat.

## Backpressure & Connection Handling

- All WebSocket sends are guarded with `ws.readyState === WebSocket.OPEN` checks
- Disconnected clients are cleaned up immediately via the `close` event
- Sessions are removed from memory on disconnect to prevent memory leaks
- A 30-second ping/pong keep-alive prevents idle connection drops
- Auto-reconnect logic in the frontend retries connection every 2 seconds on disconnect

## Setup & Running Locally

### Prerequisites
- Node.js v18+
- Supabase account
- Groq API key (free at console.groq.com)

### Steps

1. Clone the repository
2. Install dependencies:
```bash
npm install
```

3. Create `.env` file:
```env
GROQ_API_KEY=your_groq_api_key
SUPABASE_URL=your_supabase_url
SUPABASE_ANON_KEY=your_supabase_anon_key
PORT=3000
```

4. Run the Supabase SQL schema (see `/supabase/schema.sql`)

5. Start the development server:
```bash
npm run dev
```

6. Open browser at `http://localhost:3000`

## Trade-offs & Assumptions

- **Keyword RAG vs Vector RAG** — Keyword retrieval is faster and simpler for a small knowledge base. Vector embeddings would improve recall for complex queries but add latency and cost.
- **In-memory sessions** — Fast but not persistent across server restarts. A Redis cache would solve this for production.
- **Single server** — This architecture works for a single Node.js process. Horizontal scaling would require a shared session store.
- **Groq / LLaMA 3.3** — Used as a free alternative to Claude/GPT-4. Response quality is slightly lower but more than sufficient for customer support.
